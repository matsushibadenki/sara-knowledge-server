# フィードバック駆動の構造更新

## 位置づけ

RISAおよびSARA Knowledge Serverでは、フィードバックを誤差逆伝播の勾配そのものとして扱わない。

フィードバックは、上位概念・予測・文脈・評価から下位のノードや関係へ返される、**構造をどう変更するかを示す編集指令または編集候補**として扱う。

この文書は将来設計であり、現在のPhase 1実装ではまだ実行しない。

## Backpropagationとの違い

誤差逆伝播は、出力誤差から各パラメータの偏微分を計算し、勾配を入力側へ伝播させて重みを更新する。

```text
出力
→ 誤差計算
→ 各重みの偏微分
→ 入力側への勾配伝播
→ 重み更新
```

フィードバック駆動の構造更新では、全重みの勾配を計算しない。

```text
観測
→ 上位概念・予測・文脈による評価
→ 構造更新候補
→ 検証・制約確認
→ 関係・ノード・概念の局所編集
```

したがって、共通点は後段の情報が前段の状態を変えることにあるが、更新対象と計算原理は異なる。

## 生物学的フィードバックから取り入れる要素

上位の処理は、下位の処理へ次のような情報を返す。

- 予測
- 注意
- 文脈
- 期待される状態
- 予測誤差
- 構造の不整合

下位側は、入力とフィードバックを照合し、特徴・ノード・関係の解釈や活性を調整する。

```text
上位層: 「この関係が成立するはず」
        ↓ 予測・文脈
下位層: 「観測は一致するか」
        ↑ 予測誤差・反証
```

## RISAでの解釈

RISAでは、フィードバックを次の構造編集指令として表現する。

- 関係を強める
- 関係を弱める
- 関係を一時停止する
- 関係を切断する
- ノードを統合する
- ノードを分裂する
- 新しい概念を生成する
- 概念階層を変更する
- 反例または文脈限定を追加する
- 探索・活性化の優先度を変更する

```text
上位概念
→ 編集指令候補
→ 下位ノード・関係の検証
→ 局所的な構造更新
→ 更新結果を新しい観測として評価
```

## フィードバック指令の概念モデル

将来的には、以下のような共通イベントとして扱う。

```json
{
  "feedback_uid": "UUID",
  "source_type": "concept",
  "source_id": "UUID",
  "target_type": "relation",
  "target_id": "UUID",
  "operation": "strengthen",
  "magnitude": 0.2,
  "confidence": 0.8,
  "reason": {
    "prediction_error": 0.1,
    "evidence_ids": ["UUID"],
    "context": {}
  },
  "proposal_source": "risa",
  "verification_state": "proposed",
  "created_at": "2026-07-18T00:00:00Z"
}
```

初期段階では、フィードバック指令を直接適用せず、候補として保存する。適用には、権限、検証状態、対象の存在、更新制約、監査記録を必要とする。

## Knowledge Serverで保存するもの

Knowledge Serverは、RISAの編集アルゴリズムを実行する場所ではなく、構造更新に必要な履歴と根拠を保存する場所とする。

保存対象:

- フィードバック指令の候補
- 指令を生成したモデル・エンジン・バージョン
- 対象ノードまたは関係
- 予測と実観測
- 予測誤差
- 支持証拠と反例
- 適用前後の構造スナップショット
- 検証者と検証結果
- 適用・却下・取消の履歴
- 文脈、有効期間、適用範囲

将来的に専用テーブルを追加する場合も、既存の`system.audit_logs`と組み合わせ、変更前後を追跡可能にする。

## 予測符号化との関係

予測符号化から取り入れるのは、上位予測と下位観測の差分を利用して局所的に更新する考え方である。

ただし、Knowledge Serverが特定の神経科学モデルや学習則を事実として固定するわけではない。予測誤差は、関係強度、信頼度、探索優先度、レビュー優先度などを更新するための証拠の一つとして扱う。

## 安全な更新ポリシー

フィードバックによる構造変更は、次の順序で処理する。

```text
1. 候補指令を保存
2. 対象ノード・関係の存在を確認
3. 根拠と反例を確認
4. 文脈・有効期間・権限を確認
5. 変更量と制約を確認
6. 人間または承認済みルールで検証
7. 新しいバージョンまたは更新履歴を作成
8. 監査ログを保存
9. 結果をSARA/RISAへ返す
```

既存の知識を即時削除せず、強度低下、休眠、文脈限定、反例追加などの可逆的な操作を優先する。

## 評価指標

- フィードバック適用後の予測精度
- 変更されたノード・関係数
- 不要な構造変更率
- 反例を保持したままの整合性
- 更新理由の説明可能性
- 変更の再現性
- 変更の取消・復元成功率
- 人間レビュー削減量
- 構造更新あたりの計算コスト

## 設計上の結論

このプロジェクトでのフィードバック結合は、誤差逆伝播の代替実装ではない。

それは、予測・文脈・評価・反例を利用し、知識グラフやイベント構造を局所的かつ監査可能に編集するための仕組みである。

この分離により、SARAやRISAは構造を継続的に洗練でき、Knowledge Serverはその変更履歴と根拠を失わずに保持できる。
