# プロジェクト成功基準

## 一文での成功定義

AIの知識をモデルの重みから切り離し、検証・修正・共有・継承できる持続的な知識基盤を作ること。

モデルが交換されても、知識、経験、評価、関係、出典、修正履歴が失われず、再利用され続ける状態を目指す。

## 成功の本質

成功はPostgreSQLへ大量のデータを保存することではない。次の状態を実現することである。

- WordPress、LLM、SARA、RISA、学習スクリプトが共通のKnowledge Serverを利用できる
- データが出典、生成元、修正者、評価、学習利用履歴、概念、関係、反例まで追跡可能である
- 新しい情報を追加しても、システム全体を再学習せずに利用できる
- 誤った知識を修正すると、以後の利用者へ反映できる
- 同じ知識をLLM、SNN、RISAなど複数のAI方式で共有できる

## 段階別の成功

### 第一段階：データ基盤の分離

```text
AI-data-manager
＝入力・編集UI

Knowledge Server
＝データの正本
```

以下を実現する。

- Dockerで安定起動する
- API経由でデータを登録できる
- WordPressから同期できる
- データ履歴が残る
- JSONLを入出力できる
- 画像・音声などのアセットを管理できる
- データを検索できる

### 第二段階：品質管理

```text
候補
→ レビュー待ち
→ 承認
→ 学習利用
→ 評価
```

以下を分析可能にする。

- どのデータが性能改善に寄与したか
- どのデータが誤答を増やしたか
- どの生成モデルのデータ品質が高いか
- どのカテゴリーのデータが不足しているか

この段階では、データ数よりもデータの有効性を測定できることを重視する。

### 第三段階：知識変換

1つの入力から、用途の異なる成果物を生成できるようにする。

```text
文章
├── LLM用Instructionデータ
├── Q&Aデータ
├── DPOデータ
├── SARAイベント列
├── 概念
├── 実体
├── 関係
└── 出典情報
```

登録した情報を用途ごとに何度も作り直さず、Knowledge Serverを知識変換基盤として利用できる状態を目指す。

### 第四段階：経験からの構造形成

複数の観測から関係候補を形成する。

```text
ベルが鳴った
→ 食事が出た
→ ベルが鳴った
→ 食事が出た
```

この反復から、以下のような関係候補を生成する。

```text
ベル --predicts--> 食事
```

関係には証拠数、反例数、確信度、時間差、有効期間、文脈を保持する。この段階に到達すると、システムはデータ管理を超え、限定的ながら経験から構造を形成する。

### 最終段階：持続的なAI記憶

SARAや他のAIがKnowledge Serverを使うことで、モデル単体や通常のRAGよりも次の点で優れることを目指す。

- 長期的一貫性
- 知識の修正可能性
- 出典の追跡可能性
- 個人・プロジェクトへの適応
- 過去の経験の再利用
- 継続的な知識更新
- 回答根拠の説明可能性

## 比較評価

以下の3構成を同じ課題で比較する。

```text
LLM単体
LLM + 通常のRAG
LLM + SARA Knowledge Server
```

Knowledge Server構成が、正確性、一貫性、修正容易性、追跡可能性で優位になることを示す。

## 失敗の兆候

以下は成功とはみなさない。

- 大量保存できるが実際には使われない
- 関係が増えすぎて検索できない
- LLM生成の誤情報を無検証で蓄積する
- WordPressとPostgreSQLが同時に正本になる
- データ構造が複雑すぎて人間が管理できない
- 概念・関係形成のコストが効果を上回る
- 通常のRAGと性能差がない

保存量そのものを目的にせず、必要な知識を必要な場面で正しく取り出し、修正可能な形で利用できるかを評価する。

## SARAとの関係

SARAが新しいモデルへ変化しても、これまで蓄積した経験と知識を失わないことが重要な成功状態である。

研究上は、外部化された記憶と構造学習によって、モデル単体では得られない継続性を実証することを最大の成果とする。
