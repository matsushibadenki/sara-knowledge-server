# 低コスト研究・自動化戦略

この文書は、個人研究者が大規模なGPU資本に依存せず、AI-data-manager、SARA Knowledge Server、SARA、RISAを継続的に成長させるための提案を整理したものである。

以下は確定実装仕様ではなく、今後の優先順位を決めるための設計仮説・アイデアとして扱う。

## 基本戦略

大手と同じ規模のGPUを用意するのではなく、GPUを使わなくても価値が積み上がる仕組みを作る。

研究者の時間を最も高価な資源と考え、最初に自動化する対象をGPU処理ではなく、データ収集、整理、ラベル付け、品質確認、重複除去、形式変換、評価、実験管理とする。

## 3プロジェクトを循環させる

```text
AI-data-manager
        ↓
SARA Knowledge Server
        ↓
SARA / RISA
        ↓
AI-data-managerへフィードバック
```

人間が介入するのは、原則として自動処理で判断できない承認・例外確認に限定する。

## 1. 学習データ工場

AI-data-managerを単なる登録アプリではなく、データ製造ラインとして利用する。

```text
URL入力
→ スクレイピング
→ 本文抽出
→ 要約
→ Instruction生成
→ Q&A生成
→ Chat生成
→ タグ生成
→ 概念抽出
→ Knowledge Server登録
→ レビュー待ち
```

WordPressは人間による入力・承認UIとして残し、生成・解析・同期はAPIと非同期ジョブで行う。

## 2. AI同士のレビュー

OpenAI、Gemini、Qwen、Ollamaなど複数の解析器で同じ候補を評価し、合意度を品質シグナルとして利用する。

```text
複数モデルが一致
→ confidenceを上げる候補

モデル間で不一致
→ review_queueへ送る候補
```

モデルの一致だけで事実を確定してはならない。出典、人間レビュー、ルール検証、既存知識との整合性と組み合わせる。

## 3. 論理的な忘却・休眠

物理削除ではなく、利用状況と品質に基づいて検索・学習上の優先度を下げる。

評価に使う候補フィールド:

```text
use_count
last_used_at
quality_score
contradiction_count
source_count
generated_by
```

利用されないデータは`inactive`や`dormant`として扱う。ただし履歴、出典、原データは保持し、必要な場合に復元・再評価できるようにする。

## 4. 全件再学習を避ける

新規データをすべて毎回学習するのではなく、`prediction_error`、`novelty`、`coverage`、品質、既存知識との接続などから学習候補を絞る。

```text
新規データ
→ 価値評価
→ 高価値候補のみ学習キューへ
→ その他は保存・検索対象として保持
```

これはSARAのイベント学習、LLMの追加学習、Embedding生成のいずれにも適用できる考え方である。

## 5. モデルではなくKnowledge Serverを育てる

Llama、Qwen、Gemma、OpenAIなどのモデルは交換される可能性がある。一方、Knowledge Serverに蓄積した出典、履歴、評価、概念、関係は複数世代のモデルで再利用できる。

投資の中心をモデルの大型化だけに置かず、モデル非依存の知識資産、変換器、品質履歴、検索・統合機構へ置く。

## 6. Embeddingの遅延生成

初期段階から全データにEmbeddingを付与せず、まず全文検索、タグ、概念、関係を利用する。

Embeddingは次のような条件で遅延生成する。

- 実際に検索されたデータ
- 類似検索が必要なデータ
- 利用頻度が高いデータ
- 重要度・品質が高いデータ
- 明示的にEmbedding生成を要求されたデータ

これにより、Embedding API費用と保存・更新コストを抑える。

## 7. 価値の計測

データに利用価値を示す情報を蓄積する。

```text
use_count
citation_count
training_count
accepted_count
fixed_count
```

これらを、検索・学習・レビュー・出力の優先順位付けに利用する。

## 8. 夜間・低優先度バッチ

日中は登録・検索・承認などの対話的処理に集中し、計算負荷の高い処理を夜間や低優先度ジョブへ回す。

```text
Embedding生成
重複除去
関係候補生成
統計更新
品質評価
```

CPUで可能な処理を優先し、GPU・外部APIが必要な処理はまとめて実行する。

## 9. Research Queue

Knowledge Serverが、研究者が次に確認すべき課題を自動抽出する仕組みを検討する。

候補となる課題:

```text
矛盾がある
不足カテゴリがある
品質が低い
概念が孤立している
関係の証拠が弱い
レビューが滞留している
```

研究者には全データを見せるのではなく、優先順位付きの課題一覧として提示する。

## 10. 計算資源の優先順位

初期の優先順位は以下を基本案とする。

```text
★★★★★ Knowledge Server
★★★★★ 自動データ生成
★★★★★ 自動レビュー
★★★★★ データ品質管理
★★★★☆ 概念管理
★★★★☆ Relation管理
★★★☆☆ Embedding
★★☆☆☆ GPU学習
★☆☆☆☆ 巨大GPU
```

これはGPU学習を否定するものではなく、限られた資源をデータ価値の高い部分へ集中するための順序である。

## 11. 価値駆動学習

最も優先度の高い追加機能候補として、データごとの学習価値を計算する仕組みを検討する。

概念式:

```text
価値スコア
=
新規性
× 利用頻度
× 他データとの接続数
× 品質
× 将来性
÷ 学習コスト
```

実装時には、0除算、スコアの偏り、人気データへの過集中、品質と利用頻度の混同に注意する。まずは説明可能な個別要素と計算結果を保存し、後から重みを変更できるようにする。

価値スコアに基づき、次を振り分ける。

```text
高価値
→ SARA / LLMの学習キュー

中価値
→ 検索・レビュー・追加評価

低価値
→ 保存のみ、将来の再評価対象
```

## 成功を測る指標

- 人間のレビュー時間
- 1件あたりのデータ準備コスト
- 学習候補の削減率
- 学習開始までの時間
- Embedding生成件数・費用
- 自動処理から承認へ到達する割合
- AI間レビューの一致率と人間修正率
- 価値スコア上位データの性能寄与
- 夜間バッチの処理量と失敗率
- 研究者がResearch Queueで解決した課題数

## SARAの差別化

SARAの競争力を、巨大Transformerを直接上回ることだけに置かない。GPUが少なくても、昨日までの知識・経験・評価が翌日に継続利用され、研究者の作業を減らしながら知識資産が成長する「知識を捨てないAI開発環境」を目指す。
