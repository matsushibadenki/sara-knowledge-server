# SARA Knowledge Server

## PostgreSQLベース AI学習データ・イベント記憶・関係データ管理サーバー設計仕様書

バージョン: 0.1.0

# 1. プロジェクトの目的

本プロジェクトは、既存のWordPressテーマ型アプリケーション`AI-data-manager`から、AI学習データ、イベント記憶、経験、概念、関係、評価履歴、データセット、学習実験の管理機能を分離し、PostgreSQLを中心とした独立データベースサーバーとして実装する。

既存の`AI-data-manager`は、人間が学習データを入力、編集、確認、承認する管理インターフェイスとして当面維持する。

新しい`SARA Knowledge Server`は、以下の役割を担当する。

```text
・AI学習データの正本管理
・データのバージョン管理
・イベント記憶の保存
・経験単位の保存
・概念と関係の保存
・データの評価、承認、却下
・出典、ライセンス、生成元の追跡
・JSONL、JSON、CSVによる入出力
・SARA Engineへのデータ提供
・WordPressからのデータ登録
・外部学習パイプラインからのデータ取得
・将来のRISA型構造編集への対応
・ベクトル検索と全文検索
・監査ログの保存
```

本システムはLLMそのものではない。

LLM、WordPress、SARA Engine、学習スクリプトなどから利用される、独立した認知データ・学習データ基盤として設計する。

# 2. 基本方針

## 2.1 PostgreSQLを正本とする

移行完了後は、学習データ、イベント記憶、概念、関係、評価、バージョン履歴について、PostgreSQLを唯一の正本とする。

WordPressの`wp_posts`、`wp_postmeta`、独自SARAテーブルは、段階移行中のみ使用する。

最終的にWordPress側には、必要に応じて以下のみを保持する。

```text
・PostgreSQL側レコードのUUID
・表示用タイトル
・同期状態
・最終同期日時
・WordPressメディアID
```

## 2.2 UIとデータベースを分離する

管理UI、REST API、データベース、ファイル保存を分離する。

```text
WordPress AI-data-manager
Next.js管理画面
SARA Engine
Python学習スクリプト
外部LLM
        │
        ▼
SARA Knowledge API
        │
        ├── PostgreSQL
        ├── MinIO
        └── Redis
```

WordPressからPostgreSQLへ直接SQL接続してはならない。

すべての外部システムはAPIを介してアクセスする。

## 2.3 データと認知構造を論理分離する

同一PostgreSQL内で、スキーマを分離する。

```text
auth
dataset
memory
training
system
```

役割は以下とする。

```text
auth
ユーザー、APIキー、権限

dataset
学習データ、バージョン、アノテーション、出典、データセット

memory
イベント、経験、概念、実体、関係、活性、仮説

training
学習実験、モデル、チェックポイント、評価結果

system
監査ログ、ジョブ、同期状態、Webhook
```

## 2.4 すべての重要なデータにUUIDを使用する

外部公開IDにはUUIDを使用する。

PostgreSQL内部で連番IDを併用してもよいが、APIでは連番IDを公開しない。

UUID生成には`gen_random_uuid()`を使用する。

## 2.5 削除は原則として論理削除にする

学習データ、イベント、概念、関係、データセットは原則として物理削除しない。

以下の列を共通で持たせる。

```text
created_at
updated_at
deleted_at
created_by
updated_by
```

管理者向けの完全削除APIは、将来対応として分離する。

# 4. 推奨技術構成

## 4.1 APIサーバー

```text
Runtime: Bun
Language: JavaScript
Framework: Hono
Validation: Zod
Database access: Drizzle ORM
SQL migration: Drizzle Kit
API specification: OpenAPI 3.1
Authentication: JWT + API key
Password hash: Argon2id
Test: Vitest
Logging: Pino互換JSONログ
```

TypeScriptは使用せず、JavaScriptで実装する。

ただし、JSDocによる型定義を使用する。

すべてのJavaScriptファイル先頭に、配置パスをコメントとして記述する。

例:

```js
// /apps/api/src/routes/records.js
```

## 4.2 管理画面

```text
Next.js
React
JavaScript
App Router
CSS Modules
TanStack Query
React Hook Form
Zod
```

Next.jsプロジェクト作成条件:

```text
JavaScript
ESLint: Yes
Tailwind CSS: No
src directory: Yes
App Router: Yes
Turbopack: Yes
```

## 4.3 PostgreSQL

PostgreSQLは公式イメージまたはpgvector同梱イメージを使用する。

必要な拡張:

```sql
CREATE EXTENSION IF NOT EXISTS pgcrypto;
CREATE EXTENSION IF NOT EXISTS vector;
CREATE EXTENSION IF NOT EXISTS pg_trgm;
CREATE EXTENSION IF NOT EXISTS unaccent;
```

JSONB、全文検索、再帰CTE、行レベルセキュリティ、部分インデックスを活用する。

ベクトル検索は補助機能とし、明示的な関係データを置き換えない。

## 4.4 オブジェクトストレージ

画像、音声、動画、PDF、元文書などのバイナリはPostgreSQLへ直接保存しない。

MinIOへ保存し、PostgreSQLには以下を保存する。

```text
object_key
bucket_name
mime_type
size_bytes
sha256
original_filename
width
height
duration_ms
metadata
```

## 4.5 Redis

Redisは以下に使用する。

```text
・非同期インポート
・非同期エクスポート
・サムネイル生成
・Embedding生成要求
・Webhook再送
・大量データ処理
・短期キャッシュ
```

Redisを正本として使用してはならない。

## 5.2 Docker Compose要件

`docker-compose.yml`には`version`行を記述しない。

すべての永続データはnamed volumeへ保存する。

```text
postgres_data
redis_data
minio_data
```

すべての主要サービスにhealthcheckを設定する。

APIはPostgreSQL、Redis、MinIOのヘルスチェック成功後に起動する。

外部公開ポートは`.env`で変更可能にする。

初期値:

```text
Admin UI: 3000
API: 4000
PostgreSQL: 5432
MinIO API: 9000
MinIO Console: 9001
Redis: 6379
Mailpit UI: 8025
```

本番環境ではPostgreSQL、Redis、MinIO APIをホストへ直接公開しない構成を選択可能にする。

## 5.3 必須コマンド

以下のコマンドで起動できること。

```bash
docker compose up -d
```

初期化、マイグレーション、シード投入:

```bash
docker compose exec api bun run db:migrate
docker compose exec api bun run db:seed
```

テスト:

```bash
docker compose exec api bun test
docker compose exec admin npm test
```

停止:

```bash
docker compose down
```

データを含めた完全削除:

```bash
docker compose down -v
```

# 17. セキュリティ要件

```text
・パスワードはArgon2id
・JWTアクセストークンは短寿命
・Refresh Tokenはローテーション
・APIキーはハッシュ保存
・WebhookはHMAC署名
・CORS許可元を限定
・SQLは必ずパラメータ化
・アップロードMIME検査
・ファイル名をそのままobject keyに使わない
・SHA-256計算
・ファイルサイズ制限
・認証試行回数制限
・APIレート制限
・監査ログ
・個人情報をアプリログへ出さない
・秘密情報をレスポンスへ含めない
・スタックトレースを本番で返さない
```

ユーザー入力HTMLを表示する場合はサニタイズする。

MarkdownのHTML埋め込みは初期値で無効化する。

# 20. 同時編集とバージョン競合

レコード更新APIは楽観的ロックを使用する。

クライアントは更新時に現在のバージョン番号を送信する。

```json
{
  "expected_version": 4,
  "content": {}
}
```

サーバー側の最新バージョンが4でない場合は`409 VERSION_CONFLICT`を返す。

既存データを上書きしない。

管理画面では差分を表示し、以下を選べるようにする。

```text
・最新を再読み込み
・自分の変更を新バージョンとして再適用
・手動マージ
```

# 29. Codexへの実装ルール

Codexは以下を厳守すること。

```text
1. 最初に既存リポジトリ全体を確認する。
2. 既存コードを無断で大量削除しない。
3. 各フェーズを小さなコミット単位で実装する。

5. JavaScriptを使用し、TypeScriptを導入しない。
6. すべてのコードファイル先頭に配置パスを書く。
7. docker-compose.ymlにversion行を書かない。
8. 秘密情報をコードへ直書きしない。
9. SQL文字列連結を行わない。
10. 重要な更新は必ずトランザクション化する。
11. 学習データ更新時に旧バージョンを消さない。
12. APIの互換性を壊す変更を勝手に行わない。
13. API変更時はOpenAPIとdocsを同時更新する。
14. DB変更時はmigrationを追加する。
15. migration済みファイルを後から書き換えない。
16. エラーを握りつぶさない。
17. テストを追加する。
18. npmまたはbun依存追加時は理由を記録する。
19. 不要な巨大ライブラリを追加しない。
20. 実装後にREADMEを更新する。
```

部分修正を報告する際は、以下を明記する。


# 33. アーキテクチャ上の最終原則

```text
WordPress
＝人間向けの既存管理クライアント

Next.js
＝新しい標準管理クライアント

SARA Knowledge API
＝すべての読み書きの入口

PostgreSQL
＝学習データと認知構造の正本

MinIO
＝バイナリデータの正本

Redis
＝一時的なジョブとキャッシュ

SARA Engine
＝学習と推論

RISA
＝将来の構造編集・関係形成エンジン
```

データ保存層と知能アルゴリズムを分離する。

PostgreSQLには事実、仮説、履歴、関係、出典、評価を保存する。

どの知識を活性化し、どの関係を強め、何を忘れるかは、PostgreSQLの外側に置く認知エンジンが担当する。

この分離を崩さないこと。

現在のリポジトリでは、学習データが`post_content`と`post_meta`へ格納され、SARA Event Memoryのみ独自テーブル化されています。したがって、この仕様ではまず既存機能を管理UIとして維持し、独自テーブル部分からPostgreSQLへ切り出す順序にしています。

pgvectorはベクトル検索をPostgreSQLへ追加できますが、ベクトル検索だけを知識構造にせず、明示的な関係テーブルと併用する設計にしています。フィルター付き近傍検索では、データ分布や選択率によってインデックス方式と実行計画の性能が変わるため、初期段階からHNSWへ固定せず、実データで測定して導入する方針が安全です。([arxiv.org][1])

[1]: https://arxiv.org/abs/2602.11443?utm_source=chatgpt.com "Filtered Approximate Nearest Neighbor Search in Vector Databases: System Design and Performance Analysis"
