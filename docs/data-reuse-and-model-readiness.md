# データ再利用と新モデル対応

## 基本仮説

SARA Knowledge Serverは、LLMの学習アルゴリズム自体を高速化するものではない。主に、学習に使えるデータを準備する時間、品質確認のコスト、形式変換の手間を削減する。

```text
データ収集
→ 重複除去
→ 品質確認
→ 整形
→ ラベル付け
→ 出典確認
→ データセット作成
→ 学習
```

このうち学習前の工程を継続的な資産として蓄積し、新しいモデルが登場した際に再利用する。

## モデル非依存のデータ管理

データをモデル別のJSONLとして分散管理せず、Knowledge Server内に共通の内部表現を保存する。

```text
共通の内部データ
├── Instruction
├── Chat
├── DPO
├── RLHF
├── Event
├── Concept
└── Relation
```

必要な時点で、各モデルや学習フレームワークの形式へ変換する。

```text
内部形式
├── ChatML
├── ShareGPT
├── Instruction
├── DPO
├── SARA Event Sequence
└── その他の将来形式
```

重要なのは、モデルごとのデータセットを保管することではなく、共通データと再現可能な変換器を保管することである。

## 条件付きデータセット生成

出典、品質、言語、時期、タグ、検証状態などの条件を指定して、必要なデータセットを生成できるようにする。

例:

```text
承認済み
品質0.95以上
2026年以降
日本語のみ
出典あり
数学を除外
```

この条件をAPI、SQL、データセットビルダーから指定し、学習用JSONLやその他の形式へ出力する。

## 品質履歴の再利用

1件のデータに対する品質改善を、古いファイルと新しいファイルの差し替えで終わらせない。

```text
Version 1
→ Version 2
→ Version 3
```

全バージョン、変更理由、修正者、評価、承認状態を保持することで、将来の条件に応じた抽出が可能になる。

例:

```text
quality_score >= 0.9
```

という条件で、現在の高品質版だけを取得できる。

## マルチモーダル・マルチモデル対応

将来のモデルが長文、画像、音声、動画、イベントを扱う場合にも、入力の原形と構造化結果を保持しておく。

```text
画像
→ Caption / Annotation

会話
→ Chat

イベント
→ Sequence

概念・関係
→ Knowledge Graph
```

同じ知識資産から、LLM向けデータ、SARA向けイベント列、RISA向け構造化グラフを生成できる状態を目指す。

## 将来のモデル要求への適応

将来的には、モデルの仕様を入力として、要求に合うデータセットを自動生成する仕組みを検討する。

```text
モデル要件
・8万トークン対応
・画像対応
・Chain of Thought対応
        ↓
Knowledge Server
        ↓
長文・画像・推論形式に適合したデータセット
```

実現には、モデル要件の表現、変換器、Schema検証、トークン長計算、マルチモーダル整合性検査が必要になる。

## SARAとの相性

同一のデータベースから、異なる利用形式を生成する。

```text
イベント
→ 経験
→ 概念
→ 関係
→ SARA用イベント列

同じ情報
→ Instruction / Chat / DPOなどのLLM用データ
```

この構造により、LLMとSARAが同じ知識資産を異なる形式で利用できる。

## 注意点

Knowledge Serverが短縮するのは、主にデータ準備とデータセット生成である。GPUによるForward、Backward、Optimizer処理や、モデルの学習時間そのものを自動的に短縮するわけではない。

したがって、効果は以下のように分けて測定する。

- データ準備完了までの時間
- 新モデル向けデータセット生成時間
- 再利用できたデータの割合
- 形式変換エラー率
- 品質条件を満たすデータ抽出率
- 学習開始までの総リードタイム
- 学習後のモデル品質と追跡可能性

## 長期的な価値仮説

モデルは世代ごとに交換されるが、履歴、出典、評価、関係まで保持された高品質な知識資産は複数世代のモデルへ引き継げる。

このプロジェクトは単なる学習データベースではなく、モデル非依存のAI知識資産管理・変換基盤を目指す。
