# 知識獲得・構造化・統合モデル

## 位置づけ

SARA Knowledge Serverにおける「学習」は、LLMの事前学習やファインチューニングとは異なる。

LLMが大量データをトークン化し、GPU上で重みへ圧縮するのに対し、本システムは入力を受け取るたびに解析・構造化・検証し、明示的な記録としてPostgreSQLへ保存する。

基本方針は以下とする。

```text
知識の登録
→ 構造化
→ 既存知識との照合
→ 検証
→ 統合
→ SARA Engineなどへの提供
```

## 入力方法

### 人間による直接入力

Next.jsまたは既存のAI-data-managerから、タイトル、本文、データ形式、出典、タグ、品質、言語、関連概念、関係などを登録する。

Instruction、ChatML、ShareGPT、DPO、RLHF、Q&A、Plain TextなどのLLM学習用形式は、`dataset.records`として保存できる。

### ファイル・データセットのインポート

JSONL、JSON、CSV、会話ログ、Web取得データ、画像＋キャプション、センサーデータ、SARAイベント列などを扱う。

インポートデータは即時に確定知識とせず、次の段階を持つ。

```text
元データ
→ 候補知識
→ 検証
→ 承認済み知識
```

### LLMによる自動構造化

LLMは解析器・候補生成器として利用できる。文章から実体、概念、関係、イベント、時間、出典などを抽出する。

LLMの出力は原則として確定情報ではない。`proposal_source = llm`、`verification_state = unverified`などの状態で候補として保存し、人間、ルール、複数モデル、外部資料などで検証する。

### SARA・外部システムからのイベント入力

文章だけでなく、`event_type`、`modality`、`symbol`、`occurred_at`、`reward`、`prediction_error`、`state_before`、`state_after`などを持つ時系列イベントを入力する。

繰り返し観測されたイベント間の関係は、証拠数、反例数、信頼度、関係強度などを更新する候補となる。

## 3段階の処理

### 1. 取り込み

入力を失わず、原文・出典・時刻・作成者・ハッシュ・ライセンスなどを保存する。この段階では内容を確定しない。

### 2. 構造化

入力から実体、概念、イベント、関係、時間、因果、状態変化、評価などを抽出する。

### 3. 統合

既存データと照合し、同一概念か、既存関係と一致するか、矛盾か、新規情報か、例外か、出典が信頼できるかを評価する。

## LLM学習用データと知識構造の分離

同一の入力から、用途の異なる複数の成果物を生成する。

```text
元文章
├── LLM学習用レコード
├── SARAイベント記憶
└── 概念・実体・関係データ
```

LLM学習用レコードは入力と望ましい出力を表す。一方、知識構造は出典、正しさの根拠、有効期間、文脈、反例、検証者、他概念との接続を保持する。

## GPU学習との関係

知識の登録、全文検索、ベクトル検索、関係探索、ルール評価、統計更新にはGPU学習は不要である。

必要に応じて外部LLMやEmbeddingモデルを、実体抽出、概念候補生成、要約、類似・重複検出、関係候補生成、品質評価のために利用する。ただし、データベース自体を学習するのではなく、解析結果を保存する。

## 継続的な更新

新しい入力が追加されるたびに全体を再学習しない。関連候補を検索し、既存関係の`evidence_count`、`counterexample_count`、`confidence`、`strength`などを局所的に更新する。

一般的な傾向と個別の例外は削除せず、別の証拠・関係として併存させる。

## 推奨パイプライン

```text
1. WordPressまたはNext.jsからデータ登録
2. dataset.recordsへ原文を保存
3. dataset.record_versionsへ不変の版を保存
4. Redisジョブキューへ解析要求を登録
5. LLMまたはルールで構造要素を抽出
6. 候補としてmemoryスキーマへ保存
7. 既存概念・関係との重複候補を検索
8. 人間または自動ルールで検証
9. evidence_countとconfidenceを更新
10. SARA Engineへ提供
```

## システム間の責務

```text
AI-data-manager
＝データ作成、編集、アノテーション、承認

SARA Knowledge Server
＝正本保存、履歴管理、構造化、関係形成、検索、統合

SARA Engine
＝学習、推論、イベント利用
```

保存後に、WordPressまたはNext.jsからKnowledge Serverへ同期し、構造解析ジョブを登録して概念・関係候補を生成する流れを基本とする。

## 設計上の結論

本システムは、正確にはLLM学習サーバーではなく、**知識獲得・構造化・統合・検証サーバー**である。

LLM学習データ管理機能は維持するが、それと並行してイベント記憶、概念、実体、関係、証拠を管理する。
