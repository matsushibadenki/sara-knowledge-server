この文書は、2026-07-14に提供された以下のREADMEをもとに、SARA Knowledge Serverとの関係を整理したものである。

- `AI-data-manager_README.md`

## 全体像

```text
AI-data-manager
  人間向けのデータ作成・編集・アノテーション・承認UI
          │
          ▼
SARA Knowledge Server
  PostgreSQLを正本とする保存・履歴・構造化・検索・統合基盤
          │
          ├── 構造化イベント、関係、概念をRISAへ提供
          └── 疎なイベント記憶や学習データをSARA Engineへ提供
                    │
                    ▼
          SARA Engine
            CPU-firstの学習・推論・継続メモリ・エージェント基盤
```

RISAは、Knowledge Serverに保存されたイベントや関係を利用し、関係構造そのものを更新する研究コアとして位置付ける。

## AI-data-manager

WordPressを、LLM学習データ管理・アノテーション用のクローズドなWebアプリケーションとして利用するプロジェクトである。

### 主な役割

- 人間による学習データの作成、編集、確認、承認
- テキスト、画像、複雑なJSONメタデータの管理
- Plain Text、Instruction、ChatML、ShareGPT、CoT、DPO、RLHFなどの形式管理
- URLスクレイピングとLLMによるデータ生成
- データ拡張、書き換え、蒸留、Q&A生成
- JSONLインポート・エクスポート
- 外部PythonスクリプトやAxolotlなどからのREST API利用
- Bearerトークン認証による外部連携

### Knowledge Serverとの関係

AI-data-managerを廃棄せず、当面は入力・編集クライアントとして維持する。

保存操作後に、次の同期を行う。

```text
AI-data-managerで作成・編集
→ Knowledge Serverへ同期
→ record/versionとして正本保存
→ 構造解析ジョブを登録
→ 概念・実体・関係・イベント候補を生成
```

WordPressのデータをKnowledge Serverへ移行する場合、既存の投稿・メタデータ・SARA独自テーブルとのマッピングを明確にする必要がある。

## RISA

RISAは `Relationally Involving Self-organizing Architecture` の略で、重みではなく関係構造そのものを更新し続けるAIを目指す研究プロジェクトである。

### 目標

- 構造化イベント列の学習
- 動的グラフの更新
- 反復パターンからの抽象概念生成
- 次イベント予測
- 推論経路・予測理由の説明

### MVP-1

MVP-1では自然言語や画像を直接扱わず、JSON形式の構造化イベントを入力とする。

```text
イベント
→ ノード・エッジ生成
→ 反復パターンから予測関係を作成
→ 類似イベントから概念ノードを生成
→ 次のeffectを予測
→ 根拠経路を返す
```

初期実装はPythonによる小さな構成とし、データ構造の明確さ、更新追跡、説明可能性、再現性を優先する。

### Knowledge Serverとの関係

Knowledge Serverは、RISAが扱うイベント、ノード、エッジ、概念、証拠、予測結果を永続化できる基盤となる。

ただし、RISAの関係更新アルゴリズムそのものをKnowledge Serverの保存層へ混在させない。Knowledge Serverは事実、候補、証拠、履歴を保存し、RISAが構造更新・予測・探索を担当する。

## SARA Engine

SARA Engineは、CPU-firstのスパイキングAIフレームワークである。Rustによるイベント駆動コアとPythonのモデル・評価・CLI層で構成され、学習、推論、継続メモリ、軽量エージェント処理を担う。

### 主要な設計原則

- 密なテンソルではなく疎なイベントを共通インターフェースとする
- 本番ランタイムはバックプロパゲーションなし、行列処理を抑えた構成とする
- 小さく交換可能なモジュールを組み合わせる
- 実験機能はまず観測専用として評価し、確認後に実行経路へ昇格する
- スパース容量、専門サブモデル、ルーティング、コンパクトメモリで拡張する
- 重要な判断に経路証拠、反例、予測誤差、修正範囲などの説明情報を持たせる
- 生成物は`data/`、`workspace/`、`models/`などの管理ディレクトリに保存する

### 主な機能領域

- Rustアクセラレーションされたイベント駆動SNNコア
- STDP、予測符号化、FORCE、直接メモリ更新
- SNN言語・エージェント・検索・検証モジュール
- 継続メモリとリプレイ
- エネルギー効率、外部妥当性、リリースゲートの評価
- CLIによるインポート、エクスポート、学習、メモリ検査、評価

### Knowledge Serverとの関係

SARA Engineへは、Knowledge Serverから以下を提供する。

- 承認済みの学習データ
- SARAイベント列
- 状態スナップショット
- 関係・概念・実体の探索結果
- 証拠、信頼度、予測誤差、報酬などのメタデータ
- SARA Engineが生成したイベントや評価結果の取り込み先

Knowledge ServerはSARA Engineの内部メモリ実装を置き換えない。長期的な正本、履歴、出典、関係、評価を保持し、SARA Engineは実行時の学習・推論・エージェント処理を担当する。

## 3プロジェクトの責務分担

| プロジェクト | 主な責務 | 主なデータ形式 |
|---|---|---|
| AI-data-manager | 人間による入力、編集、アノテーション、承認、LLM学習データ生成 | 投稿、メタデータ、JSON、JSONL、画像 |
| SARA Knowledge Server | 正本保存、版管理、出典、構造化候補、検索、統合、監査 | PostgreSQL、JSONB、イベント、関係、アセット |
| RISA | 関係構造の更新、概念形成、次イベント予測、説明経路 | 構造化JSONイベント、動的グラフ |
| SARA Engine | 疎なイベントを使った学習、推論、継続メモリ、エージェント実行 | イベント列、状態、モデル、評価成果物 |

## 実装上の重要事項

1. AI-data-managerのLLM学習フォーマットと、RISA/SARA用のイベント・関係形式を同一視しない。
2. 同じ入力から、LLM学習用レコードと、知識構造・イベント記憶を並行して生成できるようにする。
3. LLMやRISAによる抽出結果は、まず候補・未検証データとして保存する。
4. PostgreSQLは原文、版、出典、証拠、検証状態、監査履歴を保持する。
5. RISAやSARA Engineのアルゴリズムを、Knowledge Serverの永続化層へ直接混在させない。
6. JSONLと構造化イベントJSONを、外部連携の基本形式としてサポートする。
7. SARA Engineの管理出力ポリシーに合わせ、生成物の保存場所を管理する。
8. 研究上の予測・概念形成と、確定済み知識・承認済みデータを状態として分離する。
